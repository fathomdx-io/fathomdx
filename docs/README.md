# fathomdx/docs

Canonical source for all fathomdx documentation. Markdown lives here; the site renders from these files. If it isn't in `docs/` it isn't official.

This directory uses the [Diátaxis](https://diataxis.fr) framework. Four quadrants, four different jobs. Each has its own voice and success criteria — mixing them produces docs that satisfy nobody.

## The four quadrants

| Quadrant | Job | Reader state | Voice |
|---|---|---|---|
| `tutorials/` | Teach by doing | "I'm new here" | Hand-held, single path, no branching |
| `how-to/` | Solve a named problem | "I know the basics, I need to get X done" | Goal-titled, imperative, assumes context |
| `reference/` | Look something up | "I know what I need, just tell me" | Dry, complete, no prose |
| `explanation/` | Build understanding | "I want to know why" | Discursive, conceptual, no imperatives |

If you can't place a new page in exactly one quadrant, it probably needs to be split into two.

## Writing rules

**Tutorials.** One happy path. No "alternatively," no "you could also," no footnotes. The reader follows exact steps; every command produces a visible result; each step builds on the last. Target 30–45 minutes end-to-end. Stop when the reader has a working mental model.

**How-tos.** Title starts with "How to." Assumes the reader has done the relevant tutorial (or knows equivalent). Focused on one named goal. Conditional branches are fine here — tutorials resolve "it depends"; how-tos name it.

**Reference.** Dry lookup. Tables, lists, signatures, enumerations. Almost no prose. Autogenerated where possible (HTTP API, CLI flags, env vars). Hand-written only for conventions and contracts that can't be extracted from code.

**Explanation.** Why, not how. No commands, no steps. Conceptual, discursive. Discusses trade-offs and alternatives. A reader finishing an explanation page has a clearer mental model, not a completed task.

## Audience and lingo

All docs in this directory are for **developers and self-hosting operators**. Use the developer vocabulary: **lake, deltas, sources, addons, helpers, hooks, routines, agents, plugins**.

Consumer-facing vocabulary ("mind," "moments") belongs to end-user UI copy, not these docs. When a doc references a UI element that uses consumer words (e.g., the dashboard's "Your Mind" page), name both — `` the Mind page (the lake's top-level view) `` — and link to the glossary.

See [`_glossary.md`](./_glossary.md) for the full vocabulary.

## Required frontmatter

Every page in this tree has YAML frontmatter:

```yaml
---
title: <short, human>
description: <one-line hook for site rendering and search>
audience: [developer|operator|contributor]
quadrant: [tutorial|how-to|reference|explanation]
last_verified: YYYY-MM-DD
owners: [<relative paths to code this doc describes>]
---
```

`last_verified` is a promise: on this date, a human (or agent) confirmed every command, claim, and link in this page still matches current code. Pages older than 30 days are flagged by the drift audit.

`owners` lists the files or directories whose changes should trigger a doc review. CI uses this to warn when a PR touches owned code without touching the doc.

## Drift prevention

Five mechanisms, cheapest first:

1. **Frontmatter `last_verified`.** A human or agent sets this only after confirming the page is accurate. No handwaving.
2. **Tutorials are executable.** Every code block in `tutorials/` is either pulled from a real file (`addons/scripts/install.sh` etc.) or runs in the install-smoke CI job. If the command breaks, the tutorial breaks.
3. **Reference is autogenerated.** HTTP API captured from FastAPI's `/docs`. Env vars grepped from code. CLI from `--help`. Regeneration runs in CI; drift produces a diff the PR author must address.
4. **Link and command linter.** A CI script parses all markdown, follows internal links, and checks that referenced files, flags, and env vars exist. Catches ~80% of rot mechanically.
5. **Monthly drift audit.** A scheduled agent walks every page with `last_verified` older than 30 days, compares against current code, and opens a PR with findings.

See [`STATUS.md`](./STATUS.md) for the current page inventory and verification state.

## Adding a page

1. Decide the quadrant. If you're unsure, it's probably two pages.
2. Create the file in the matching directory.
3. Fill in required frontmatter.
4. Follow the voice rules for that quadrant.
5. Add an entry to `STATUS.md`.
6. Link from adjacent pages where relevant. Don't create orphans.

## What's not here

- `../QUICKSTART.md` stays at the repo root. It's the 5-minute front door; the tree here is what you read when QUICKSTART isn't enough.
- `../CLAUDE.md` is agent-facing conventions, not user docs.
- Per-addon READMEs stay with their addons. The docs tree links to them; it doesn't duplicate them.
